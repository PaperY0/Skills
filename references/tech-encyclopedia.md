# 技术栈百科

每学到一个技术时，不仅讲"怎么用"，还要讲"为什么它存在"、"它和相近技术的区别"、"企业为什么选它"。

---

## React

### 一句话概括
React 是一套"乐高积木"——把页面拆成独立的小组件，改一块不影响另一块。

### 为什么 React 火了？
- **组件化**：每个按钮、卡片、输入框都是独立组件。一个地方写，到处能用。
- **虚拟 DOM**：React 在内存里算好变化再一次性更新页面，比直接操作浏览器 DOM 快。
- **生态最大**：npm 上 React 相关的包最多，招人也最容易。

### React vs Vue
| | React | Vue |
|---|-------|-----|
| 写法 | JSX（HTML 写在 JS 里） | 模板（HTML + 指令） |
| 学习曲线 | 较陡（需要理解 JSX、Hooks） | 较平缓（模板语法接近 HTML） |
| 企业采用 | 北美/国内大厂主流 | 国内中小企业较多 |
| 灵活性 | 高（只有 UI 层，其他自己选） | 中（官方提供路由、状态管理等） |

### 新手应该先学哪个？
如果你目标是进大厂 → React。如果你只想快速做项目 → Vue。但从 PRE 的角度，学 React 能让你更深入理解"组件化"和"数据流"。

## TypeScript

### 一句话概括
TypeScript 是"带合同的 JavaScript"——提前约定好每个人传什么数据，防止传错。

### 为什么企业喜欢 TypeScript？
- **提前发现 Bug**：不用等到上线才炸。类型不对当场报错。
- **AI 友好的基础设施**：AI 返回的数据用 TS 类型约束，格式错了编译不过。
- **代码即文档**：看到类型定义就知道这个函数要什么参数、返回什么。

### TypeScript vs JavaScript
就像"签了合同的快递" vs "口头约定的快递"。
- JS：你说"给我寄个东西"→ 寄了个砖头你也得收
- TS：你说"寄一个盒子，里面是书，不能是砖头"→ 寄砖头当场拒收

## Vite

### 一句话概括
Vite 是"自动化工厂流水线"——你把源代码放进去，它自动打包成浏览器能跑的文件。

### Vite vs Webpack
- Vite 启动 <1秒，Webpack 启动可能需要几十秒
- Vite 用浏览器原生 ES Module，开发时不打包，改一行只更新那一行
- 2024 年以后新项目基本都选 Vite

## Zustand

### 一句话概括
Zustand 是"微信群公告栏"——你在一个地方改了数据，所有关注这个数据的地方自动看到。

### 为什么选 Zustand 而不是 Redux？
| | Zustand | Redux |
|---|---------|-------|
| 代码量 | 10 行搞定 | 需要 action/reducer/store 三板斧 |
| 学习曲线 | 几乎不用学 | 需要理解 reducer、middleware |
| TypeScript 支持 | 原生支持 | 需要额外配置 |

### 什么时候不用 Zustand？
- 如果你的数据只需要在一个页面里用，useState 就够了
- 如果只有父子组件用，props 就够了
- 不是所有数据都需要全局化

## IndexedDB + Dexie.js

### 一句话概括
IndexedDB 是"浏览器自带的大仓库"，能存海量结构化数据。Dexie 是让这个仓库更好用的管理系统。

### IndexedDB vs localStorage
| | IndexedDB | localStorage |
|---|-----------|-------------|
| 容量 | 几百MB~几GB | 5-10MB |
| 数据类型 | 对象、数组、文件 | 只能字符串 |
| 查询 | 支持索引、排序、过滤 | 只能全读出来自己找 |
| 异步 | 是（不卡页面） | 否（同步操作） |

### 为什么不用 localStorage？
localStorage 就像一个"小抽屉"，只能放几张小纸条。IndexedDB 就像"仓库"，能放成百上千箱货。

## Repository 模式

### 一句话概括
Repository 是"仓库管理员"——你告诉他"存这个"、"取那个"，他帮你操作数据库，你不需要知道数据库怎么开门关门。

### 为什么不能直接操作数据库？
```
直接操作：                通过 Repository：
Page1 → DB              Page1 → Repo → DB
Page2 → DB              Page2 → Repo → DB
Page3 → DB              Page3 → Repo → DB
...                     ...
Page99 → DB             Page99 → Repo → DB

❌ 99处直接操作             ✅ 99处通过同一个管理员
改一个字段→改99处           改一个逻辑→只改 Repository 一处
```

### 企业什么时候加更多层？
- 小项目：Page → DB（能跑）
- 中型项目：Page → Repository → DB
- 大项目：Page → Service → Repository → DAO → DB
- 每一层都解决"代码放一起太难改"的问题

## Prompt 工程

### 为什么 Prompt 不写在前端？
- API Key 会被看到（F12 就能翻）→ Key 泄露被刷爆
- Prompt 模板在前端容易被复制拿走
- 后端可以用缓存、限流、日志等企业级能力

### 为什么 AI 返回要用 Zod 校验？
AI 不是数据库——它可能返回格式不对、少字段、多字段、用错词。Zod 是"合同验收员"，检查 AI 返回的东西是否符合约定。

## Promise / async / await

### 生活类比
- **Promise**：快递单号。快递还没到，但你知道"会到的"
- **await**：等快递到了再拆。没到之前先干别的
- **async**：标记"这个流程里可以等快递"

### 为什么需要异步？
浏览器只有一个"主线程"——如果等数据时页面卡住不动，用户体验极差。异步让页面在你等数据时还能操作。
